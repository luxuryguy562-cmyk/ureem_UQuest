# PROJECT_CONSTITUTION.md

# U-Quest 개발 헌법

이 문서는 U-Quest 프로젝트의 최상위 운영 규칙이다.  
모든 에이전트, 개발자, AI 도구, 코덱스 작업은 이 문서를 우선 기준으로 삼아야 한다.

---

# 1. 최상위 원칙

## 1.1 문서 우선 원칙

U-Quest는 문서를 기준으로 개발한다.

기능, 화면, DB, 경제식, 관리자 정책, API, UX 흐름이 변경될 경우 반드시 관련 `.md` 문서를 먼저 업데이트하거나 동시에 업데이트해야 한다.

문서가 업데이트되지 않은 변경은 완료된 작업으로 보지 않는다.

---

## 1.2 최신 문서 기준 원칙

모든 작업자는 작업 시작 전 다음 문서를 확인해야 한다.

- PRD.md
- CTO_AGENT.md
- SYSTEM_ARCHITECTURE.md
- DATABASE_SCHEMA.md
- DB_DESIGN_DRAFT.md
- ECONOMY_RULE.md
- ADMIN_RULE.md
- TESTING_AND_SIMULATION.md
- FRONTEND_GUIDE.md
- DESIGN_RULE.md
- API_SPEC.md
- GAME_DESIGN.md
- AGENT_RULE.md
- WORK_LOG.md

과거 대화나 기억보다 최신 `.md` 문서를 우선한다.

`uquest_master_documentation_v_1.md`는 초기 합본 원본으로 보관한다.  
실제 작업 기준은 위 개별 문서이며, 합본과 개별 문서가 다르면 개별 문서를 우선한다.

---

## 1.3 쉬운 설명 원칙

프로젝트 오너는 코딩/개발 전문 지식이 많지 않다.

따라서 모든 에이전트와 개발자는 설명할 때 다음 원칙을 지켜야 한다.

- 어려운 기술어를 먼저 쓰지 않는다.
- 기술어를 써야 한다면 쉬운 비유를 함께 쓴다.
- “왜 필요한지”를 먼저 설명한다.
- 선택지가 있다면 장단점을 비교해서 추천한다.
- 최종 추천안을 명확히 말한다.
- 사용설명서처럼 설명한다.

예:

나쁜 설명:
“RLS 정책과 Edge Function으로 처리하면 됩니다.”

좋은 설명:
“직원마다 자기 데이터만 보게 막는 안전장치가 필요합니다. Supabase에서는 이걸 RLS라고 부릅니다. 쉽게 말하면 직원별 출입증 같은 역할입니다.”

## 1.4 DESIGN_RULE 준수 원칙

모든 화면/UI/목업/모바일 작업은 DESIGN_RULE.md를 반드시 따른다.

Design Agent는:
- 모바일 360px 기준
- 콘텐츠 잘림 방지
- ellipsis 최후 사용
- HTML 목업 우선
- 실측 폭 표 작성
- 스켈레톤 우선 UX
- 정보 보존 레이아웃

원칙을 준수해야 한다.

Frontend Agent는 DESIGN_RULE.md를 기준으로 실제 구현한다.

---

## 1.5 범용 UX는 발명하지 않는 원칙

출석, 캘린더, 요일, 검색, 정렬, 필터, 탭, 토글, 장바구니, 알림, 승인, 설정 저장 같은 범용 앱 기능은 새로 창조하지 않는다.

이미 대부분의 사용자가 익숙한 앱 문법을 우선한다.

예:

- 출석체크는 실제 날짜/요일 칸에 반영한다.
- 2026년 6월 1일 월요일에 출석하면 `월` 칸이 오늘/완료 상태가 되어야 한다.
- `월 화 수 목 금 토 오늘`처럼 요일 자리에 임의의 `오늘` 칸을 추가하지 않는다.
- 캘린더는 실제 달력 구조를 따른다.
- 정렬/검색/필터는 일반적인 업무 도구 패턴을 따른다.

U-Quest만의 창의성은 세계관, 성장감, 보상 연출, 사내 문화 맥락에 사용한다. 기본 조작 문법은 사용자가 이미 아는 방식을 따른다.

# 4. Git / PR / 롤백 규칙

## 4.1 메인 브랜치 직접 수정 금지

`main` 브랜치에 직접 작업하지 않는다.

모든 작업은 별도 브랜치에서 진행한다.

예:

```text
feature/home-screen
feature/economy-engine
fix/calendar-layout
admin/reward-settings
```

---

## 4.2 PR 필수

작업 완료 후 반드시 Pull Request를 만든다.

PR에는 다음 내용을 포함해야 한다.

```md
## 작업 내용
- 무엇을 변경했는지

## 확인 방법
- 어떻게 테스트하면 되는지

## 영향 범위
- 사용자 화면
- 관리자 화면
- DB
- API
- 경제식

## 롤백 방법
- 문제 발생 시 되돌릴 방법
```

---

## 4.3 롤백 가능 상태 유지

모든 PR은 문제가 생기면 즉시 이전 상태로 되돌릴 수 있어야 한다.

금지:
- 여러 기능을 한 PR에 섞기
- DB 변경과 UI 변경을 설명 없이 섞기
- 경제식 변경을 문서 없이 반영하기
- 되돌릴 수 없는 직접 수정

# 5. 에이전트 역할 구조

U-Quest는 단일 에이전트가 모든 것을 결정하지 않는다.  
각 에이전트는 역할을 나누어 검토한다.

---

## 5.1 CTO Agent

### 역할
프로젝트 전체 기술 방향을 관리한다.

### 책임
- 기술스택 추천
- 전체 구조 검토
- 프론트/백엔드/API 연결 방향 검토
- 확장성 검토
- 개발 난이도 조절
- 오너에게 쉬운 설명 제공
- 각 에이전트 작업 결과 검수
- 문서/코드/DB/API/경제식 충돌 확인
- 작업 완료 전 CTO 최종 보고 작성

### 필수 태도
- 사용설명서처럼 설명한다.
- 기술적으로 더 좋은 방향이 있으면 먼저 추천한다.
- 오너가 모르는 부분은 쉽게 풀어준다.
- 어려운 말은 쉬운 비유로 바꿔 설명한다.

### 필수 준수 문서
- CTO_AGENT.md

# 7. 경제 계산 원칙

## 7.1 타격권은 줄이지 않는다

타격권은 재미와 참여감을 담당한다.

예산이 초과될 경우 타격권을 줄이는 것이 아니라 다음 항목을 조정한다.

- 1타 평균 코인
- 히든 확률
- 이벤트 배율
- 경제보정계수

---

## 7.2 성장과 현금성 보상을 분리한다

성장 요소:
- SXP
- 사람 레벨
- 캐릭터 외형
- 갑옷
- 칭호
- 배지
- 프로필 프레임
- 검 외형

현금성 보상:
- 코인
- 상품권
- 히든 보상
- 기프티콘

성장은 크게 제공하고, 현금성 보상은 예산 안에서 통제한다.

---

## 7.3 코인 정산 기준

MVP의 운영/정산 기준은 다음과 같다.

```text
1코인 = 1원 상당
```

단, 코인은 현금이 아니다. 코인은 앱 안에서 보상으로만 교환되는 내부 포인트다.

이 기준을 쓰는 이유:

- 오너와 관리자가 월 예산을 직관적으로 이해할 수 있다.
- 보상 가격을 실제 상품권 가격과 맞춰 관리하기 쉽다.
- 경제 시뮬레이션에서 예상 지급액을 바로 계산할 수 있다.

코인 단위 변경은 경제식, 상점 가격, 관리자 예산 화면을 동시에 바꾸는 큰 변경으로 취급한다.

---

## 7.4 성장 레벨 상한

사람 레벨과 검 레벨은 분리한다.

- 사람 레벨은 SXP 누적으로만 오른다.
- 검 레벨은 SXP 해금 조건을 만족한 뒤 골드로 강화한다.
- 검 강화는 사람 레벨을 직접 올리지 않는다.
- 검 레벨은 MVP에서 코인 획득량, 히든 확률, 타격권 수량에 영향을 주지 않는다.
- MVP 검 레벨 상한은 Lv.10이다.
- Lv.10 이후는 숫자 레벨을 계속 늘리지 않고 숙련도, 외형, 배지, 시즌 티어로 확장한다.
- 주문서는 MVP에서 비활성화한다. 추후 필요하면 랜덤 드랍이 아니라 고정 미션/관리자 지급으로만 검토한다.

레벨은 무한히 큰 숫자를 만들기 위한 장치가 아니라 온보딩 단계와 성장 체감을 보여주는 장치다.

---

# 8. 관리자 우선 원칙

관리자 페이지에서 조정 가능한 항목은 실제 사용자 화면에 즉시 반영되어야 한다.

예:
- 미션 추가 → 홈 온보딩에 반영
- 보상 추가 → 상점에 반영
- sort_order 변경 → 카드 순서 반영
- 보상설정 변경 → 나무/검/히든 확률 반영
- 직원 승인 → 로그인 가능 여부 반영

# 9. 작업 시작 전 체크리스트

모든 작업자는 작업 시작 전 확인한다.

```md
- [ ] 관련 최신 .md 문서를 읽었는가?
- [ ] CTO_AGENT.md 기준으로 검수 대상이 있는가?
- [ ] 이번 작업이 어떤 문서에 영향을 주는가?
- [ ] DB 변경이 필요한가?
- [ ] 관리자 화면 변경이 필요한가?
- [ ] 사용자 화면에 반영되는가?
- [ ] 경제식에 영향이 있는가?
- [ ] 롤백 가능한 작업인가?
```

# 10. 작업 완료 체크리스트

```md
- [ ] 코드 변경 완료
- [ ] 관련 문서 업데이트 완료
- [ ] WORK_LOG.md 업데이트 완료
- [ ] PR 생성
- [ ] 테스트 방법 작성
- [ ] 롤백 방법 작성
- [ ] 오너에게 쉬운 설명 작성

```

# 11. 최종 원칙

```text
문서가 기준이다.
작업은 기록한다.
PR 없이 main에 넣지 않는다.
문서 없는 변경은 완료가 아니다.
경제식은 서버에서 계산한다.
관리자 설정이 사용자 화면을 지배한다.
오너에게는 사용설명서처럼 쉽게 설명한다.
```
