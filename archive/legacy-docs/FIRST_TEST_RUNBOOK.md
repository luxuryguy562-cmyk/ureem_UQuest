# U-Quest 첫 테스트 실행 순서

## 1. 지금 방향

첫 테스트의 목적은 "게임이 재미있는가"보다 "신입사원이 온보딩 미션을 자연스럽게 수행하고 진행 상태를 이해하는가"를 확인하는 것이다.

테스트 핵심 흐름:

```text
출석
→ 미션 완료
→ 포인트/경험치 지급 확인
→ 챕터 진행 확인
→ 배지/칭호/프레임 확인
→ 포인트로 쿠폰 교환 확인
```

나무 타격, 타격권, 검 강화는 최신 MVP 테스트 대상이 아니다.

## 3. 첫 DB 연결 순서

1. Supabase 환경변수가 `.env.local`에 있는지 확인한다.
2. `.env.local`은 GitHub에 올라가지 않아야 한다.
3. `app_config_snapshots.current.payload`를 읽어 화면이 뜨는지 확인한다.
4. 관리자 미션 저장 API가 스냅샷을 갱신하는지 확인한다.
5. 정식 운영 전에는 운영 테이블 저장과 스냅샷 재발행 구조를 분리한다.

## 4. 오너가 테스트할 때 볼 것

홈:

- 오늘 해야 할 일이 바로 보이는가
- 현재 챕터/진행 상태가 이해되는가
- 포인트와 경험치가 섞여 보이지 않는가

미션:

- 미션을 왜 해야 하는지 이해되는가
- 완료 시 포인트/경험치 변화가 보이는가
- 관리자 승인 미션과 자동 완료 미션이 구분되는가

챕터:

- 2개월 온보딩 여정이 보이는가
- 지금 어느 단계인지 이해되는가
- 다음 단계로 가기 위해 무엇을 해야 하는지 보이는가

컬렉션:

- 획득한 배지/칭호/프레임이 보이는가
- 대표 장착이 이해되는가
- 아직 못 얻은 항목의 조건이 과하게 복잡하지 않은가

보상:

- 보유 포인트가 보이는가
- 쿠폰 가격이 명확한가
- 교환 후 포인트 차감과 쿠폰함 반영이 맞는가

---

## 5. 합격 기준

- 사용자가 "오늘 뭘 해야 하는지" 5초 안에 이해한다.
- 포인트는 쿠폰용, 경험치는 성장용이라는 구분이 된다.
- 챕터 진행이 온보딩 여정처럼 느껴진다.
- 배지/칭호/프레임이 동료에게 보여줄 만한 자랑 요소로 느껴진다.
- 불필요한 중간 클릭 없이 보상이 반영된다.
