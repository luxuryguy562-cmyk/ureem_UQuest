# 1. 최상위 원칙

## 1.1 온보딩 우선 원칙

U-Quest는 게임 앱이 아니라 온보딩 앱이다. 게임 요소는 신입사원이 온보딩 미션을 더 쉽게 수행하고, 진행 상태를 더 즐겁게 이해하도록 돕는 장치다.

아래 방향은 금지 또는 보류한다.

- 나무 타격 중심 구조
- 타격권 중심 경제
- 검 강화 중심 성장
- 채굴/룰렛/슬롯형 보상
- 과도한 전투형 RPG 구조

---

## 1.2 문서 우선 원칙

기능을 바꾸기 전에 관련 문서를 먼저 바꾼다.

우선순위:

1. `PRD.md`
2. `GAME_DESIGN.md`
3. `ECONOMY_RULE.md`
4. `ADMIN_RULE.md`
5. `DATABASE_SCHEMA.md`
6. `API_SPEC.md`
7. `FRONTEND_GUIDE.md`
8. `DESIGN_RULE.md`

문서와 코드가 충돌하면 최신 변경 이력이 있는 개별 문서를 우선한다.

---

## 1.3 쉬운 설명 원칙

오너에게 설명할 때는 기술 용어보다 제품 흐름을 먼저 말한다.

좋은 설명:

```text
미션을 완료하면 포인트와 경험치가 들어가고,
경험치는 챕터와 캐릭터 진행도를 올리고,
포인트는 쿠폰 교환에만 씁니다.
```

나쁜 설명:

```text
서버 함수가 ledger와 snapshot payload를 갱신합니다.
```

---

## 1.4 범용 UX 원칙

출석, 캘린더, 요일, 검색, 정렬, 필터, 탭, 토글, 승인, 설정 저장 같은 범용 기능은 일반 앱 문법을 따른다.

U-Quest만의 특수 문법을 새로 만들지 않는다.

# 2. 문서 관리 규칙

## 2.1 문서 업데이트 필수 상황

아래 변경이 있으면 관련 문서를 함께 업데이트한다.

- 서비스 방향 변경
- 핵심 루프 변경
- 화면/UX 변경
- DB 구조 변경
- 경제식/보상/포인트 정책 변경
- 관리자 기능 변경
- API 변경
- 에이전트/작업 규칙 변경

모든 큰 작업은 `WORK_LOG.md` 상단에 기록한다.

형식:

```text

### 작업 요약
### 변경 파일
### 결정 사항
### 검증
### 남은 문제

```

# 3. Git / PR / 롤백 규칙

## 3.1 메인 브랜치 직접 수정 금지

실제 운영 단계에서는 main 브랜치 직접 수정을 금지한다.

## 3.2 PR 필수

운영 배포 전에는 PR로 변경 내용을 검토한다.

PR에는 아래 내용을 포함한다.

- 작업 내용
- 확인 방법
- 영향 범위
- 롤백 방법

## 3.3 롤백 가능 상태 유지

DB migration, API 변경, 포인트 지급 로직 변경은 반드시 롤백 계획을 가진다.

# 6. 경제 계산 원칙

## 6.1 포인트는 경제다

포인트는 쿠폰 교환용 실질 재화다. 예산 통제 대상이다.

## 6.2 경험치는 성장이다

경험치는 챕터, 레벨, 캐릭터 진행도에 쓰인다. 쿠폰 교환에 사용하지 않는다.

## 6.3 명예 보상은 자랑이다

배지, 칭호, 프로필 프레임은 동료 간 자랑 요소다. 예산을 직접 소모하지 않는다.

## 6.4 쿠폰은 서버에서 처리한다

쿠폰 교환은 포인트 차감, 재고 차감, 쿠폰 발급, 인벤토리 추가가 원자적으로 처리되어야 한다.
