# U-Quest 첫 테스트 실행 순서

이 문서는 코딩을 잘 몰라도 같은 순서로 테스트할 수 있게 만든 운영용 메모입니다.

## 1. 지금 방향

지금은 완벽한 DB를 먼저 만드는 단계가 아니라, 화면 버튼 기준으로 한 명의 신입이 처음부터 끝까지 움직이는 흐름을 먼저 살리는 단계입니다.

첫 기준 흐름:

1. 김은성 신입 1일차 화면 진입
2. 오늘 출석하기
3. 미션 1개 완료
4. 타격권 지급 확인
5. 나무 치기
6. 코인/히든/주문서 지급 확인
7. 관리자 대시보드에서 직원 상태 확인

이 흐름이 DB까지 연결되면 그 다음부터 보상액, 확률, 미션 구성, 관리자 수정 화면을 깎아가면 됩니다.

## 2. 로컬 서버 켜기

PowerShell에서 프로젝트 폴더로 이동한 뒤 실행합니다.

```powershell

powershell -ExecutionPolicy Bypass -File .\scripts\start-local-dev.ps1
```

브라우저 주소:

```text
http://127.0.0.1:3000
```

서버 끄기:

```powershell
cd C:\Users\es\Documents\Codex\U-Quest
powershell -ExecutionPolicy Bypass -File .\scripts\stop-local-dev.ps1
```

## 3. 로컬 서버가 닫히는 이유

`127.0.0.1:3000`은 내 컴퓨터 안에서만 열리는 개발 서버입니다.

닫히는 대표 이유:

- 터미널을 닫았다.
- Codex/VS Code/PowerShell을 재실행했다.
- 컴퓨터가 절전/재부팅됐다.
- 코드 런타임 에러로 개발 서버가 불안정해졌다.
- 3000 포트를 쓰던 프로세스가 종료됐다.

그래서 로컬 개발은 계속 켜줘야 하는 임시 작업장이고, 외부에서 안정적으로 보려면 배포 주소가 필요합니다.

## 4. GitHub + Cloudflare 배포 판단

추천은 둘 다 쓰는 방식입니다.

로컬:

- 코덱스가 빠르게 수정하고 바로 확인하는 작업장
- 버튼/DB 로직 개발에 적합

GitHub + Cloudflare:

- 꺼지지 않는 테스트 URL
- 휴대폰이나 다른 사람에게 공유 가능
- 커밋을 올릴 때마다 자동 배포 가능

단, DB 로직이 붙는 U-Quest는 단순 정적 HTML보다 서버 동작이 중요해질 수 있습니다. 그 경우 Cloudflare Pages 정적 배포보다 Cloudflare Workers/Next.js 방식이나 Vercel 배포가 더 맞을 수 있습니다.

## 5. 첫 DB 연결 순서

첫 DB 테스트는 이 순서로 작게 시작합니다.

1. `profiles` 또는 테스트 사용자 기준으로 김은성 1명 상태를 읽는다.
2. `claim_attendance` 함수로 오늘 출석을 처리한다.
3. `complete_mission` 함수로 미션 완료와 타격권 지급을 처리한다.
4. `hit_tree` 함수로 타격권 차감과 코인 지급을 처리한다.
5. 화면에서 결과를 다시 읽어 반영한다.
6. 관리자 화면에서 같은 결과를 확인한다.

중요 원칙:

- 버튼 하나는 DB 함수 하나로 연결한다.
- 재화 증감은 직접 업데이트보다 원장 기록을 남긴다.
- 테스트 데이터는 김은성 1명부터 시작한다.
- 잘 돌아가는 첫 루프를 만든 뒤 보상/확률/관리자 기능을 넓힌다.

## 6. 오너가 테스트할 때 볼 것

출석하기:

- 오늘 출석 상태가 완료로 바뀌는가
- 타격권이 증가하는가

미션 완료:

- 체크 표시가 생기는가
- 같은 미션을 두 번 눌렀을 때 중복 지급이 막히는가

나무 치기:

- 타격권이 1개 줄어드는가
- 코인이 증가하는가
- 획득 내역이 보이는가

관리자:

- 김은성 상태가 바뀌는가
- 미션 추가가 사용자 화면에 반영되는가
- 보상 설정 숫자를 바꿀 수 있는 구조로 이어지는가
