# 1. 테스트를 보는 큰 그림

U-Quest 테스트는 5단계로 본다.

| 단계 | 확인하는 것 | 쉬운 설명 |
|---|---|---|
| 1. DB 구조 테스트 | 테이블, 관계, RLS, 정책 | 건물 골조와 출입문이 제대로 생겼는지 확인 |
| 2. Seed 테스트 | 초기 미션/보상/검/경제 설정 | 빈 가게에 상품과 가격표를 제대로 진열했는지 확인 |
| 3. 경제 시뮬레이션 테스트 | 예산, 코인, 히든 확률 | 손님이 많이 와도 돈이 새지 않는지 계산 |
| 4. 기능 흐름 테스트 | 출석, 미션, 타격, 강화, 교환 | 직원이 실제로 눌렀을 때 기록과 잔액이 맞는지 확인 |
| 5. 화면 테스트 | 관리자 설정이 사용자 화면에 반영되는지 | 사장이 가격을 바꾸면 진열대 가격표도 바뀌는지 확인 |

# 3. 다음에 할 Seed 테스트

오너가 직원/미션/보상 양식을 주면 아래 순서로 seed를 넣는다.

1. 회사/매장/팀

5. 미션 그룹/미션

| 확인 항목 | 기대값 |
|---|---|
| 미션 개수 | 양식에 적힌 개수와 일치 |

| 프론트 화면 | fallback이 아니라 Supabase 값을 읽음 |

# 4. 기능 흐름 테스트

나중에 서버 함수가 붙으면 아래 흐름을 테스트한다.

## 4.1 출석

```text
출석 버튼
→ attendance_logs 생성
→ wallet_ledger에 ticket 지급 기록
→ sxp_ledger에 SXP 지급 기록
→ user_wallets 잔액 증가
```

합격 기준:

- 같은 날짜에 중복 출석이 안 된다.
- 타격권과 SXP가 원장에 남는다.
- 현재 잔액과 원장 합계가 맞는다.

## 4.2 미션 완료

```text
미션 완료
→ mission_completions 생성
→ 자동 승인 또는 관리자 승인
→ 타격권/SXP 지급
```

합격 기준:

- 일일 제한을 넘지 않는다.
- 승인 전/후 상태가 구분된다.
- 같은 요청이 두 번 들어와도 중복 지급되지 않는다.

## 4.3 나무 타격

```text
나무 타격
→ 타격권 1 차감
→ tree_hit_logs 생성
→ 코인/SXP/히든 결과 지급
```

합격 기준:

- 타격권이 없으면 실패한다.
- 코인 지급량이 경제 스냅샷 기준을 따른다.
- 히든 당첨 여부와 확률이 로그에 남는다.

## 4.4 검 강화

```text
검 강화
→ 코인 차감
→ user_sword_states 갱신
→ sword_upgrade_logs 생성
```

합격 기준:

- 재료가 부족하면 실패한다.
- 성공 시 현재 검 레벨이 바뀐다.
- 차감 원장이 남는다.

## 4.5 보상 교환

```text
상점 교환
→ 코인 차감
→ reward_redemptions 생성
→ inventory_items 생성
```

합격 기준:

- 재고가 없으면 실패한다.
- 코인이 부족하면 실패한다.
- 교환 기록과 보상함이 함께 생긴다.

# 5. RLS 보안 테스트

RLS는 직원별 출입증 같은 역할이다.

확인할 것:

| 사용자 | 볼 수 있어야 하는 것 | 못 봐야 하는 것 |
|---|---|---|
| 직원 | 본인 지갑/미션/보상함 | 다른 직원 지갑/로그 |
| 점장 | 자기 매장 직원 | 다른 매장 직원 |
| 팀장 | 자기 팀 직원 | 다른 팀 직원 |
| 슈퍼관리자 | 전체 | 없음 |

이 테스트는 seed 데이터가 들어간 뒤 실행한다.

# 6. 보상 시뮬레이션을 확인하는 방법

오너가 확인해야 하는 핵심 숫자는 6개다.

| 항목 | 의미 |
|---|---|
| 월 예상 타격수 | 직원들이 한 달에 나무를 몇 번 칠지 |
| 1타 평균 코인 | 한 번 칠 때 평균 몇 코인을 줄지 |
| 예상 월 지급액 | 이번 설정으로 한 달에 얼마가 나갈지 |
| 경제보정계수 | 예산 초과를 막기 위해 보상을 얼마나 줄이는지 |
| 히든 확률 | 히든 보상이 나올 확률 |
| 예산 위험도 | 안정/주의/위험 |

관리자 화면에서는 아래처럼 보여야 한다.

```text
입력값:
- 활성 직원 수
- 참여율
- 평균 일일 타격권
- 1인 월 기본 보상 한도
- 1인 월 최대 보상 한도
- 코인 원가
- 히든 1회 평균 가치
- 이벤트 배율
- 평균 검 배율

결과:
- 월 예상 타격수
- 월 기본 예산
- 월 히든 예산
- 1타 평균 코인
- 예상 월 지급액
- 경제보정계수
- 히든 확률
- 예산 위험도
```

---

# 7. 오너가 보는 합격 기준

보상 시뮬레이션은 아래 기준으로 판단한다.

| 상태 | 기준 | 해석 |
|---|---|---|
| 안정 | 예상 지급액이 목표 예산의 90% 이하 | 운영 여유 있음 |
| 주의 | 예상 지급액이 목표 예산의 90~100% | 조정 필요 가능성 있음 |
| 위험 | 예상 지급액이 목표 예산 초과 | 코인/확률/배율 조정 필요 |

추가로 꼭 봐야 하는 것:

- 참여율 100%일 때도 예산이 터지지 않는가?
- 이벤트 배율 2배일 때도 위험하지 않은가?
- 검이 성장해도 지급액이 폭증하지 않는가?
- 히든 확률이 너무 높지 않은가?
- 1인 월 평균 보상이 회사가 감당 가능한 범위인가?

# 8. 현재 사용할 수 있는 로컬 시뮬레이터

아직 실제 seed 데이터가 없으므로, 지금은 샘플 숫자로 경제 공식을 확인한다.

실행:

```bash
npm run simulate:economy
```

결과는 터미널에 다음처럼 나온다.

```text
월 예상 타격수
1타 평균 코인
예상 월 지급액
경제보정계수
히든 확률
예산 위험도
```

나중에는 이 계산을 관리자 화면과 `economy_snapshots` 테이블에 연결한다.

---

# 9. 최종 운영 방식

오너가 최종적으로 확인하는 방식은 이렇다.

1. 관리자 화면에서 경제 설정을 입력한다.
2. "시뮬레이션" 버튼을 누른다.
3. 예상 지급액과 위험도를 본다.
4. 안정이면 저장한다.
5. 저장하면 `economy_snapshots`와 `app_config_snapshots`가 갱신된다.
6. 사용자 화면은 새 설정을 읽는다.

즉, 오너가 SQL이나 코드를 직접 볼 필요는 없다.  
최종 제품에서는 관리자 화면의 시뮬레이션 카드만 보면 된다.
